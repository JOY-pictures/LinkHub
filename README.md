# 📘 LINKHUB — КОНЦЕПТУАЛЬНЫЙ ФУНДАМЕНТ

## 1. Главная идея

LinkHub — это социальная сеть, где:

> Пользователь управляет лентой, а не алгоритм управляет пользователем.

Платформа ориентирована на осознанное потребление контента, а не на максимизацию удержания.

---

# 2. Основная философия

### 2.1 Контроль пользователя

* Лента конечна.
* Пользователь сам выбирает лимит постов.
* Пользователь выбирает режим (настроение / фокус).
* Режим запоминается.

### 2.2 Осознанный дизайн

* Нет бесконечного скролла.
* Нет давления популярности.
* Количество жалоб не отображается.
* Нет соревнования за статус.

### 2.3 Прозрачность

Каждый пост может быть объяснён кратко и конкретно:

* почему он показан,
* какие факторы повлияли,
* как текущий режим влияет на ленту.

---

# 3. Режимы ленты (User Mode)

Режим — это переключатель фокуса пользователя.

Примеры режимов:

* Balanced (по умолчанию)
* Learning
* Fun
* Calm
* Deep Thinking
* Motivation

## Важные принципы:

* Режим влияет значительно.
* Режим важнее прошлых привычек.
* Прошлое поведение — вспомогательный фактор.
* Пользователь всегда видит текущий режим.
* Если лимит ленты маленький — показывается только чистый режим.
* При большом лимите в конце может появляться небольшой “расширяющий” блок.

---

# 4. Структура ленты

Лента состоит из:

### 4.1 Основной части

* Контент максимально соответствует режиму.
* Учитываются свежесть, реакции, поведение пользователя.

### 4.2 Расширяющего хвоста (опционально)

* Небольшой процент контента вне привычных паттернов.
* Добавляется только при большом лимите.
* Нужен для предотвращения пузыря.

---

# 5. Сигналы интереса

Система учитывает:

1. Недавние реакции пользователя.
2. Тип/характер контента.
3. Реакции других пользователей.
4. Просмотр поста пользователем (если без реакции).

Важно:

* Интересы со временем “устаревают”.
* Недавнее важнее старого.

---

# 6. Многомерные реакции

Вместо лайков используются смысловые реакции:

* Useful
* Inspiring
* Insightful
* Funny
* Supportive

### Принципы:

* Реакции отражают “за что ценят”.
* В MVP можно ограничиться одной реакцией на пост.
* В будущем возможно несколько реакций.

---

# 7. Классификация контента

Автор при публикации указывает, какому режиму соответствует пост.

⚠️ Важно:
Это не "тип поста", а "контекст публикации".

Пост — это универсальная единица контента.
Режим — это метка, отражающая намерение автора.

Это нужно для:

* работы режимов с первого дня,
* быстрого MVP,
* осознанной маркировки.

---

# 8. Доверие автора

Вводится динамический показатель доверия.

## 8.1 Смысл доверия

Это не рейтинг популярности.
Это показатель точности классификации.

## 8.2 Механизм

* Пользователи могут сообщить о несоответствии режима.
* Учитываются только массовые устойчивые расхождения.
* Жалобы не публичны.
* Хейт не должен влиять.

## 8.3 Последствия снижения доверия

* Снижается влияние выбранного автором типа.
* Система больше опирается на реакции аудитории.
* Это не бан и не наказание за непопулярность.

## 8.4 Видимость

* Показатель видит только сам автор.
* Отображается в его “студии”.
* Не используется как публичный статус.

---

# 9. Правила ленты

1. Лента конечна.
2. Лимит выбирает пользователь.
3. Смешиваются свежие и ценные старые посты.
4. Два поста одного автора подряд не показываются.
5. Режим запоминается.
6. Режим влияет значительно.
7. Баланс: режим > прошлое поведение.
8. При большом лимите возможен расширяющий блок.
9. Каждый пост можно кратко объяснить.

---

# 10. Персонализация

* Учитывается прошлое поведение.
* Но пользователь может “переключить сознание” через режим.
* Система не запирает человека в пузыре.
* Персонализация — мягкая и управляемая.

---

# 11. Архитектурный принцип

Система строится так, чтобы:

* Можно было добавлять режимы без переписывания старого.
* Можно было усиливать алгоритм.
* Можно было заменить внутреннюю доставку событий.
* Можно было масштабировать.
* Бизнес-логика не зависела от Telegram или конкретного интерфейса.

---

# 12. MVP (минимальный жизнеспособный продукт)

На старте:

* Публичные текстовые посты.
* Режимы ленты.
* Конечная лента с лимитом.
* Многомерные реакции.
* Базовая персонализация.
* Объяснение “почему показано”.
* Доверие автора.

Без:

* Дополнительных типов постов (музыка, видео, пины и т.д.)
* Сложных настроек приватности.
* Социального рейтинга.

---

# 13. Главный принцип системы

LinkHub — это не машина удержания.
Это инструмент управления вниманием.

14. Универсальная модель поста

Пост в LinkHub — это не конкретный тип контента, а универсальная единица публикации.

Пост представляет собой контейнер, который может содержать различные формы контента.

Важно:

Пост ≠ текст

Текст — это один из вариантов поста

Архитектура не должна предполагать, что пост может быть только текстовым

15. Типы постов (расширяемая модель)

Система должна поддерживать возможность добавления различных типов постов без изменения существующей логики.

Примеры типов:

TEXT_POST — текстовый пост (MVP)

PIN_POST — визуальный вдохновляющий пост (аналог Pinterest)

MUSIC_RELEASE — музыкальный релиз

PLAYLIST_POST — плейлист

PODCAST_POST — аудио-контент

VIDEO_POST — видео

Принцип:

Добавление нового типа поста должно происходить через расширение, а не изменение существующего кода.

16. Независимость ленты от типа контента

Алгоритм ленты работает на уровне поста, а не конкретного типа контента.

Это означает:

Лента не знает, текст это или музыка

Лента работает с "единицей публикации"

Тип контента не влияет на архитектуру ленты

Следствие:

Можно добавлять новые типы контента без переписывания алгоритма.

17. Разделение понятий: тип поста и режим

В системе существуют два разных уровня:

Тип поста

Определяет форму контента:

текст

музыка

видео

изображение

Режим (User Mode)

Определяет:

в каком контексте показывать контент

какое настроение или цель у пользователя

Важно:

Тип поста ≠ режим

Один и тот же тип поста может подходить разным режимам

Режим влияет на отбор, а не на структуру

18. Долгосрочное развитие платформы

При сохранении текущих архитектурных принципов LinkHub может эволюционировать в:

социальную сеть

контент-платформу

систему рекомендаций

музыкальный стриминговый сервис

библиотеку медиа

платформу авторов

Важно:

Это не цель MVP.

Это потенциал системы, который должен учитываться при проектировании.

19. Ограничение сложности MVP

Несмотря на широкую архитектуру, MVP должен оставаться простым.

Правило:

Мы проектируем систему с учётом будущего, но реализуем только необходимое.

На старте:

только текстовые посты

простая модель

минимум сущностей

20. Ключевой архитектурный принцип

Пост — это универсальная единица публикации.
Тип контента является расширением поста, а не его заменой.
